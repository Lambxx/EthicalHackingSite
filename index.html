A guide on XSS

What is XSS

XSS stands for cross-site scripting, a broad category of vulnerabilities focusing on code injection that causes code to run on a target's web application. It is one of the most prolific vulnerabilities on the web, though in recent years, with increased filtering, simple XSS vulnerabilities have been reduced. However, many more complex vulnerabilities are still out there, and the impact can be critical. When your browser loads a web page (like this one ), it performs a request to the server hosting the website at the address you have given, and the server responds with the associated resources for this webpage [1]. Generally speaking, this will consist of some HTML (The content of the page), some CSS (The styling for the page), and some JavaScript (This handles dynamic and interactive elements); these are done by your browser locally. XSS aims to inject some code the browser processes (the payload) with the rest of the code and run this on the victim's web browser.

Types of XSS

There are four main categories of XSS

Reflected XSS

Reflected XSS uses an HTTP URL (the website address) and corresponding request, which is sent when you request that URL, to inject the payload onto the site. Reflected XSS is called such because the payload is injected on the victim's URL, sent through an HTTP request and immediately reflected through the HTTP response onto the page[2]. It, therefore, requires a victim to inject the payload on their machine, which means reflected XSS usually requires malicious links and social engineering to be effective. For example, a hacker could send a malicious link through a phishing email. When the user clicks the link, the payload that was placed in the URL runs on the site the hacker specified in the link.

Stored XSS

In contrast, stored XSS does not require a victim to pollute their URL, which can make it a more critical vulnerability. Stored XSS relies upon the payload being stored persistently by the web application and sent to victims via a later HTTP request[3]. The script is executed once the payload arrives on the victim's web browser. Suppose a user is on a social media site, and suppose said media site does not have adequate protection, then some user can comment a payload on the website. Doing so causes the payload to be stored in the server/database and displayed to every user who looks at said comment and executes the payload on their browser.

One of my first interactions with cybersecurity was hearing the story of Sammy the Worm in my teens, a famous stored XSS attack that caused chaos on Myspace. You can hear the story from the hacker Samy Kamkar here.

DOM XSS

Both reflected and stored XSS rely on the server passing some payload to the browser; in the case of reflected, it is immediately reflected from the URL, while stored, it is fetched from some persistent location. On the other hand, DOM-based XSS does not rely on the payload passing through the server at all[4]. Instead, DOM-based XSS uses the Javascript document object Model[4]. DOM allows programs to dynamically change the web page's content [5]. DOM-based XSS exploits this by passing a malicious payload through what is referred to as the 'source' to a vulnerable function called a 'sink', which allows execution of the payload(4). You can read about common sinks here.To give an example, imagine a website, which displays the value you have searched for at the top of the page when you press a button. This search input would be the source; the function that takes the input and displays it without the proper precautions( this is expanded on below) is the sink, and the payload will then execute on the browser.

Blind XSS

With a blind XSS attack, you are attempting to attack a target that is not immediately visible. This vulnerability focuses on submitting an XSS payload, hoping it will trigger on some invisible system, such as an admin page. For example, I could make my email an XSS payload, hoping it is processed somewhere else where it will execute; tools can be used to find out if the payload does execute and where.

Impact of XSS

Stealing cookies

Cookies are a popular way to maintain sessions and authenticate user requests [6]. XSS can be used as a technique to steal these cookies. Once the cookies are stolen, they can be injected back into the browser and impersonate the victim. [7]The impact of impersonation depends on the application, but this could be writing defamatory posts from a social account or reading sensitive data.

Password autofill

XSS can exploit users with auto-fill password managers by allowing the password manager to populate some password field, which then sends the user password to the attacker[7].

CSRF

Whilst the above two impacts used to be an issue with increased prevention measures recently, this is not so much the case. Whilst these vulnerabilities have reduced, more complex layered issues still exist; CSRF is an example of this. CSRF is a separate vulnerability from XSS; CSRF uses a user being logged in somewhere else to perform actions on behalf of their account. CSRF is protected against using a CSRF token. However, XSS can bypass this. [7] an XSS vulnerability opens the door to another vulnerability.

Causes of XSS and how we can prevent it

Improper input validation

Not validating inputs can leave room for XSS vulnerabilities; for example, an input expecting an age should not allow a string; by stopping this, we prevent potential payloads from ever being processed. Ideally, these inputs should be blocked from ever being submitted[8] .

Not encoding, sanitising and escaping unsafe characters

An unsafe character may be processed in some unintended way, such as a URL or a code segment; examples include ""<",".>" {","}", "|", "\", "^", "~", "[","]", and" `"." [9] . Unsafe characters should be sanitised, encoded or escaped. Sanitisation involves removing unsafe characters entirely, often by allowing characters that match a safelist or disallowing characters that match a denylist. [10] . Escaping involves transforming the input so it is no longer unsafe, for example, adding \ before a " so that it is not read as closing a string but just as text [11]. Encoding involves transforming the characters to an equivalent encoded character that is not processed unsafely by the web application; for example, < is encoded as < This is also used when displaying HTML code on a page as a code snippet, so it is not read as HTML, the example html given in the demonstrations is encoded in this way

Exploring further we can find a blog post.

This seems a good potential location to inject a payload.

We will attempt to inject a basic payload, between script tags, which if successful will show an alert of 1, to anyone who views the comment. We fill out the rest of the form with valid values so it is accepted. Were this to be unsuccessful we could try placing the payload in other felids, such as name.

It is however successful and after submitting the comment we are returned to the blog, where the payload is executed on our browser, this would now be the case for anyone who views the blog post.

Once we close the alert, we can see the blog post and comments, looking at the comments we can see our comment there. It does not show the payload as it has been processed as code and not text to be displayed.

However inspecting the code for the webpage, allows us to see our script has been placed between the p tags, where the comment should have gone.
